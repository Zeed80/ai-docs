# AI Docs

> Статус: проект находится на стадии активной разработки и пока не является работоспособным продуктом для реального использования.

AI Docs - локальная платформа для обработки технических, закупочных и учетных документов с AI-памятью, нормоконтролем, графом связей и встроенным инженерным агентом.

Проект ориентирован на производственные задачи: счета и закупки, документы НТД, техпроцессы, оснастку, инструмент, станки, складские операции, нормы и инженерные взаимосвязи между документами. Конфиденциальные OCR, извлечение данных и анализ по умолчанию рассчитаны на локальную инфраструктуру.

## Возможности

- Загрузка документов через веб-интерфейс и API.
- OCR и извлечение текста из PDF, изображений и офисных документов.
- Классификация документов и постановка задач на асинхронную обработку.
- Управление загруженными документами: просмотр, выборочное удаление, массовое удаление, очистка dev-данных.
- SQL-ориентированная память документов для быстрого поиска по тексту и метаданным.
- Векторная память и rerank/embedding-профили с пользовательским выбором моделей.
- Построение графа связей после загрузки документов.
- Поиск и редактирование зависимостей между документами.
- Нормоконтроль НТД в автоматическом режиме или по кнопке пользователя.
- Встроенный AI-агент для инженерных, технологических, складских и закупочных сценариев.
- Черновики писем, согласования, аудит действий и approval gates.
- Интеграционный слой OpenClaw подготовлен как опциональное направление, но основной рабочий агент сейчас встроен в проект.

## Архитектура

- `backend/` - FastAPI, SQLAlchemy, Alembic, Celery, OCR/extraction, AI router, API.
- `frontend/` - Next.js, React, интерфейсы документов, настроек, согласований и рабочих разделов.
- `openclaw/` - промпты, сценарии, skill registry и конфигурации интеграции.
- `infra/` - Docker Compose, Traefik и инфраструктурные настройки.
- `docs/` - планы, аудит, статус рефакторинга и регрессионные манифесты.
- `scripts/` - проверки, генераторы registry и benchmark tooling.

Инфраструктурные сервисы:

- PostgreSQL - основная SQL-база и быстрый текстовый слой.
- Redis - брокер Celery и кэш.
- MinIO - объектное хранилище исходных файлов.
- Qdrant - векторное хранилище.
- Ollama - локальные LLM, OCR/vision, embeddings и reasoning-модели.
- Celery - фоновые задачи распознавания, классификации, извлечения, embedding и графа.

## Требования

- Linux, macOS или Windows с WSL2.
- Docker и Docker Compose.
- Node.js 20+ для локальной разработки frontend без Docker.
- Python 3.11+ для локальной разработки backend без Docker.
- Ollama, если планируется локальный AI вне контейнера.
- GPU желателен для OCR/vision и больших локальных моделей.

Рекомендуемые локальные модели задаются в настройках проекта и `.env`:

- `OLLAMA_MODEL_OCR=gemma4:e4b`
- `OLLAMA_MODEL_REASONING=gemma4:26b`

Для embeddings и rerankers проект должен использовать выбранный пользователем профиль модели; параметры подтягиваются через registry и возможности провайдера.

## Быстрый старт

```bash
git clone https://github.com/Zeed80/ai-docs.git
cd ai-docs
cp .env.example .env
```

Проверьте `.env` перед запуском:

- замените `APP_SECRET_KEY`, `POSTGRES_PASSWORD`, `MINIO_SECRET_KEY`;
- настройте `OLLAMA_URL`;
- для доступа с другого компьютера в сети задайте `NEXT_PUBLIC_API_URL` и `NEXT_PUBLIC_WS_URL` с IP сервера;
- не добавляйте реальные ключи и пароли в git.

Запуск dev-стека:

```bash
make dev
```

Альтернативно напрямую через Docker Compose:

```bash
docker compose -f infra/docker-compose.yml -f infra/docker-compose.dev.yml up
```

Основные адреса по умолчанию:

- Frontend: `http://localhost:3000`
- Backend API: `http://localhost:8000`
- OpenAPI: `http://localhost:8000/docs`
- MinIO Console: `http://localhost:9001`
- Qdrant: `http://localhost:6333`
- Ollama: `http://localhost:11434`

Если frontend открывается с другого компьютера в локальной сети, в `.env` укажите IP машины с backend:

```env
NEXT_PUBLIC_API_URL=http://192.168.1.10:8000
NEXT_PUBLIC_WS_URL=ws://192.168.1.10:8000
```

## Работа с документами

1. Загрузите документ в разделе документов или inbox.
2. Система сохранит исходный файл, создаст задачу обработки и запустит OCR/classification/extraction.
3. После обработки документ появится в управлении документами с типом, статусом, извлечениями и связями.
4. Зависимости можно искать, просматривать и редактировать.
5. Удаление документа должно очищать исходный файл, SQL-записи, связанные извлечения, graph/memory-записи и векторные данные best-effort.

В настройках разработки есть полная очистка документов и записей БД. Это временная dev-функция, перед production ее нужно защитить или убрать.

## Нормоконтроль НТД

Нормоконтроль поддерживает два режима:

- автоматическая проверка после обработки документа;
- ручная проверка по кнопке `Проверить на соответствие НТД`.

Цель модуля - проверять документы на соответствие нормативно-технической документации, фиксировать замечания, связывать их с источниками НТД и сохранять результат в аудитируемой истории.

## AI-агент

Встроенный агент должен работать как самостоятельный инженерный сотрудник: технолог, конструктор, нормировщик, снабженец, кладовщик и помощник по документообороту. Для внешних действий, отправки писем, удаления данных и других рискованных операций сохраняются approval gates и аудит.

Основные правила:

- конфиденциальные документы обрабатываются локально;
- пользователь выбирает модели и провайдеры в настройках;
- агент использует SQL-память, векторный поиск, rerankers и граф связей;
- подтверждения пользователя постепенно сокращаются только для безопасных и хорошо проверенных операций.

## OpenClaw

В проекте есть директория `openclaw/` с локальными сценариями, промптами и конфигурациями. Официальный open-source OpenClaw пока оставлен как опциональная интеграция и не является обязательной для запуска основного приложения.

Локальные runtime-файлы, device identity, токены, логи и скачанные зависимости OpenClaw не публикуются в репозитории.

## Безопасность

- Не коммитьте `.env`, реальные документы, базы данных, дампы, токены и production credentials.
- Для production обязательно смените все значения `changeme`.
- Включайте auth/SSO и ограничивайте CORS.
- Проверяйте политику хранения документов и журналов аудита.
- Dev-endpoint полной очистки данных не должен быть доступен без защиты в production.
