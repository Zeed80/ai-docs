# Рефакторинг проекта счетов в AI-сотрудника документооборота

## Что представляет из себя проект

Проект по счетам должен развиваться не как “OCR для PDF”, а как специализированное рабочее место AI-сотрудника для учета, закупки, склада и финансового контроля. Его основная задача — принимать входящие счета, КП, акты, накладные и письма от поставщиков, понимать их содержание, проверять риски, готовить действия для человека и вести прозрачную историю решений.

Если AI Manufacturing Workspace — это широкий инженерно-производственный cockpit для технолога, конструктора, нормировщика, учетчика и кладовщика, то данный проект — его узкая, прикладная версия вокруг финансово-закупочного документооборота. Он должен быть “умным сотрудником по счетам”: не просто распознавать текст, а понимать контекст поставщика, историю цен, дубли, реквизиты, сроки, складские последствия и согласования.

Система должна работать по принципу human-in-the-loop: AI предлагает, объясняет и подготавливает, но внешние действия выполняются только после approval. Отправка писем, экспорт в 1С, подтверждение оплаты, изменение справочников и принятие подозрительных документов должны проходить через явное согласование и audit.

## Главная роль AI-сотрудника

AI-сотрудник в этом проекте выполняет роль внимательного помощника учетчика, снабженца и кладовщика:

- принимает входящие счета и документы;
- определяет тип документа;
- извлекает реквизиты, суммы, НДС, строки, позиции, сроки оплаты и поставки;
- сверяет счет с историей поставщика;
- проверяет арифметику и дубли;
- находит изменения реквизитов;
- замечает рост цен и нетипичные позиции;
- готовит карточку аномалий;
- предлагает письмо поставщику или внутреннему сотруднику;
- готовит экспорт в Excel или 1С;
- ведет audit timeline;
- не делает внешних действий без разрешения человека.

Идеальная формулировка продукта: это не “сервис распознавания счетов”, а “электронный сотрудник, который разбирает входящие счета, подсказывает риски и готовит учетные действия под контролем человека”.

## Что проект будет уметь

### 1. Входящие документы

Проект должен принимать документы из нескольких источников:

- ручная загрузка PDF/JPG/PNG/XLSX/CSV/XML/JSON;
- email inbox через IMAP;
- вложения из писем;
- drag-and-drop в интерфейсе;
- будущие интеграции с 1С, ЭДО, CRM или файловыми папками.

Каждый входящий файл должен становиться объектом `Document` со статусом, hash, источником, связью с поставщиком/счетом и audit-событиями.

Подозрительные файлы не должны обрабатываться автоматически. Для них нужен quarantine:

- extension allowlist;
- статус `suspicious`;
- запрет processing до ручной проверки;
- audit event;
- будущий antivirus/mime scan.

### 2. Распознавание и структурированное извлечение

Система должна извлекать данные не в свободный текст, а в строгие Pydantic-схемы:

- поставщик;
- ИНН/КПП;
- банковские реквизиты;
- номер счета;
- дата счета;
- валюта;
- сумма без НДС;
- НДС;
- сумма с НДС;
- строки счета;
- артикулы;
- наименования;
- количество;
- единицы измерения;
- цена за единицу;
- сумма строки;
- срок оплаты;
- срок поставки;
- комментарии и условия.

Для каждого поля нужно хранить:

- значение;
- confidence;
- reason;
- источник;
- страницу или bbox, если доступно.

AI-вызовы должны идти только через единый AI-router, чтобы можно было менять Ollama/vLLM/другие модели без переписывания бизнес-логики.

### 3. Проверка счетов строгим кодом

ИИ не должен быть единственным источником истины. Все критические проверки должны выполняться детерминированным кодом:

- `quantity * unit_price == line_total`;
- сумма строк равна subtotal;
- subtotal + tax == total;
- НДС соответствует ставке;
- номер счета не дублируется у этого поставщика;
- hash файла не совпадает с уже загруженным;
- сумма и номер не повторяют старый счет;
- дата счета не выглядит ошибочной;
- реквизиты поставщика не изменились без причины.

Результат проверки должен сохраняться как structured check result, а не только как текстовый комментарий.

### 4. Поставщики и история цен

Проект должен вести базовый профиль поставщика:

- название;
- ИНН;
- КПП;
- банковские реквизиты;
- контактные лица;
- email-домены;
- история счетов;
- история реквизитов;
- история цен по позициям;
- trust score;
- открытые вопросы и спорные документы.

История цен нужна не как отчет “когда-нибудь”, а как ежедневный инструмент:

- подсветить рост цены выше порога;
- найти нехарактерную цену;
- сравнить несколько КП;
- показать последнюю закупку такой позиции;
- предложить, у кого раньше покупали дешевле;
- сформировать вопрос поставщику.

### 5. Аномалии и риск-карточки

Для каждого счета система должна уметь создавать `AnomalyCard`.

Типовые аномалии:

- дубликат счета;
- изменились реквизиты;
- цена выросла;
- сумма не сходится;
- НДС не сходится;
- поставщик новый или неизвестный;
- файл подозрительного типа;
- слишком низкая confidence у ключевых полей;
- номер счета похож на старый, но сумма другая;
- в письме указан один поставщик, а в счете другой;
- счет пришел с подозрительного email-домена.

Карточка аномалии должна содержать:

- severity;
- список сигналов;
- объяснение;
- рекомендуемое действие;
- требуется ли approval;
- ссылки на документ, поставщика и строки счета.

### 6. Email workspace

Проект должен работать с письмами не как с “сырой почтой”, а как с частью документооборота.

Функции:

- принять письмо;
- связать его со счетом/поставщиком;
- сохранить вложения как документы;
- распознать контекст письма;
- предложить ответ;
- проверить риски перед отправкой;
- заблокировать отправку до approval;
- сохранить всю переписку в audit timeline.

AI-сотрудник должен уметь готовить письма:

- запросить недостающие реквизиты;
- уточнить срок поставки;
- попросить корректный счет;
- сообщить о расхождении суммы;
- запросить КП;
- попросить подтвердить изменение цены;
- подготовить внутреннее письмо на согласование.

Отправка письма должна быть внешним действием и всегда проходить через approval gate.

### 7. Экспорт и интеграции

Проект должен готовить данные для учетных систем, но не отправлять их бесконтрольно.

Минимальные функции:

- Excel export;
- JSON payload для 1С;
- подготовка проводки/документа поступления;
- audit события экспорта;
- approval gate перед 1С export;
- signed file URLs для скачивания файлов;
- будущий adapter для ЭДО.

Важно: “подготовить экспорт” и “отправить во внешнюю систему” — разные действия. Первое может быть безопасным, второе требует approval.

## Основные рабочие сценарии

### Smart ingest счета

1. В систему попадает PDF/JPG/XLSX счет.
2. Файл проходит quarantine/allowlist.
3. Pipeline извлекает текст, preview и OCR при необходимости.
4. AI-router извлекает structured invoice.
5. Backend проверяет арифметику и дубли.
6. Система обновляет поставщика и историю цен.
7. Если есть риск, создается `AnomalyCard`.
8. Пользователь видит документ в review-интерфейсе.
9. После проверки данные можно экспортировать в Excel или подготовить 1С payload.

### Проверка входящего письма

1. IMAP adapter забирает письмо.
2. Вложения становятся документами.
3. Письмо связывается с поставщиком.
4. AI определяет, что нужно сделать.
5. Если не хватает данных, создается черновик ответа.
6. Risk-check проверяет получателей, суммы, вложения и реквизиты.
7. Отправка блокируется до approval.

### Сравнение КП

1. Пользователь загружает несколько КП.
2. Система извлекает позиции.
3. Позиции нормализуются к canonical item.
4. UI показывает side-by-side compare.
5. История цен подсвечивает лучший вариант.
6. AI готовит краткое объяснение выбора.
7. Решение пользователя сохраняется в audit.

### Работа с аномалией

1. Система создает `AnomalyCard`.
2. Пользователь открывает причину.
3. Видит source evidence: поле, строка, документ, прошлый счет.
4. Принимает решение: approve, reject, request clarification.
5. Если нужен поставщик, AI готовит письмо.
6. Все действия сохраняются в audit timeline.

## Интерфейс

Интерфейс должен быть не “табличкой счетов”, а cockpit для ежедневной работы.

Ключевые экраны:

- “Моя работа сегодня”;
- Inbox входящих документов;
- карточка счета;
- side-by-side review: документ слева, поля справа;
- поставщик и история цен;
- anomaly card;
- письма и черновики;
- approval queue;
- audit timeline;
- Excel/1C export panel.

UX-принципы:

- keyboard-first;
- быстрый triage;
- фокус на полях с низкой confidence;
- видимые причины решений AI;
- никаких скрытых внешних действий;
- command palette `Ctrl+K`;
- статусы видны сразу;
- audit понятен человеку, а не только разработчику.

## Архитектурные принципы

### Backend

FastAPI должен быть источником истины:

- модели БД;
- Pydantic-схемы;
- REST API;
- document processing;
- invoice checks;
- audit;
- approval gates;
- exports.

### AI layer

AI слой должен быть модельно-независимым:

- registry моделей;
- task routing;
- local-only policy;
- structured outputs;
- валидация Pydantic;
- fallback;
- запрет прямых вызовов Ollama/vLLM из бизнес-кода.

### Agent layer

Agent/OpenClaw должен быть “мозгом”, но не бесконтрольными руками:

- tools генерируются из OpenAPI/Pydantic;
- allowlist tools;
- unknown tools denied;
- step limits;
- approval gates;
- audit каждого шага;
- external actions blocked until approval.

### Security

По умолчанию:

- confidential/local-only;
- signed file URLs;
- quarantine;
- RBAC;
- OIDC;
- audit;
- encrypted backups;
- production migrations через Alembic;
- никаких silent production `create_all`.

## Что важно не делать

- Не делать “чат поверх PDF” вместо структурированной системы.
- Не хранить результат только в `ai_summary`.
- Не доверять арифметику LLM.
- Не отправлять письма или 1С payload без approval.
- Не завязывать бизнес-логику напрямую на Ollama/vLLM.
- Не смешивать confidential документы с cloud AI без явного разрешения.
- Не делать UI, где оператор обязан читать весь PDF вручную каждый раз.
- Не скрывать причины низкой confidence.

## Итоговое позиционирование

Проект должен стать не узким OCR-инструментом, а надежным AI-сотрудником для счетов и закупочного документооборота. Он берет на себя рутинную работу: разобрать документы, найти риски, сверить суммы, сравнить историю, подготовить письма и экспорт. Человек остается владельцем решения: утверждает, исправляет, отклоняет и видит всю историю.

Главная ценность проекта — снижение ручной нагрузки и ошибок без потери контроля.
