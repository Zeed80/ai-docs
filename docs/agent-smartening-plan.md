# План усиления агента: «делает что просят, помнит, учится»

Цель: агент понимает задачу и публикует на Рабочий стол **именно то, что просили**;
при неуверенности ведёт себя **адаптивно по риску** (дешёвое/обратимое — делает с
явными допущениями; дорогое/внешнее — спрашивает); и **постоянно учится** на
успехах и на правках пользователя.

Принцип: **усиливаем существующее, не строим с нуля.** Базис уже есть:
- Гейт качества: `backend/app/ai/audit.py` (`AuditCode`, `RETRYABLE`) + post-run
  `_audit_turn` / retry-петля `_build_correction_request` в `orchestrator.py`.
- Семантический критик: `semantic_passed` / `AuditCode.SEMANTIC_SUSPECT`
  (бюджет `aux_quality_budget`).
- Обучение: `orchestrator_memory.py` (фидбэк-память, успех skills),
  `recipes.py` (draft→active replay), `skill_evolver.py`.
- Риск-классы: 9 approval gates (`email.send`, `invoice.approve`, …) = «дорогое».
- Spec-таблицы: `domain/table_spec.py`, публикация `api/spec_tables.py`.

Каждая фаза: реализация → пересборка прод-стека → **живая проверка** реальными
сценариями (как фрезы/резцы) → unit-тесты → метрика до/после.

---

## Фаза 1 — Гейт «делает что просят» (correctness), адаптивный по риску

Проблема: после построения spec-таблицы агент публикует её без сверки **содержимого**
с запросом. Аудит сейчас проверяет фильтры/канвас/пустоту, но не «отвечает ли
таблица на вопрос» (тот ли источник, колонки, группировка).

1. **Intent-match проверка опубликованного артефакта.** В `_audit_turn` добавить
   проверку: сверить опубликованный `TableSpec` (source/columns/filters/group_by)
   и краткую сводку результата с намерением запроса. Новый `AuditCode.INTENT_MISMATCH`
   (или усиление `SEMANTIC_SUSPECT`, чтобы критик смотрел на саму таблицу, а не
   только на фильтры). Дёшево: детерминированные сигналы (пустой результат при
   непустом запросе, источник не из запроса, отсутствует запрошенная группировка)
   + один бюджетный LLM-критик на спорных случаях.
2. **Адаптивное поведение по риску** (новый хелпер `risk_class(plan/action)`):
   - дёшево/обратимо (desktop spec-table): при mismatch — retry-петля (есть), а
     если после ретраев всё ещё спорно — публикует лучший вариант + короткая
     приписка с **явными допущениями** и chip «уточнить» (вариант «сделать +
     показать допущения»).
   - дорого/внешнее (approval-gated): **никогда не публиковать молча** — pre-action
     intent-match, при сомнении спросить ДО действия.
3. Метрика: доля ходов, где артефакт соответствует намерению (выборка ручной
   разметки + авто-сигналы); падение «опубликовал не то».

Файлы: `app/ai/audit.py`, `app/ai/orchestrator.py` (`_audit_turn`,
`_build_correction_request`, новый `risk_class`), тесты `tests/test_orchestrator*`.

## Фаза 2 — Понимание задачи (router → spec)

Проблема: воркер строит спеку, не совпадающую с намерением (слепые дженерик-таблицы,
неверный источник/колонки), и не уточняет неоднозначное.

1. **Уточнение неоднозначного, адаптивно.** Когда `TurnDecision.confidence` низок
   ИЛИ слоты неоднозначны (какие именно фрезы / какой поставщик / какой период):
   дёшево — построить лучший вариант + допущения; при высокой неоднозначности +
   дорогом действии — короткий уточняющий вопрос ДО построения.
2. **Жёстче привязка к каталогу.** Усилить grounding воркера на
   `spec_table_catalog` (источники/поля/синонимы из `table_spec.SOURCES`), чтобы
   source/columns/filters выбирались из whitelisted-каталога, а не выдумывались.
3. Расширить детерминированный `reconcile_ops`/`parse_patch_command` на больше
   формулировок (группировка/сортировка/сравнение/период), 0 LLM.
4. Метрика: меньше audit-ретраев и пустых/mismatch-результатов на наборе реальных
   запросов.

Файлы: `app/ai/turn_router.py`, `app/ai/orchestrator.py`, `app/domain/table_spec.py`.

## Фаза 3 — Постоянное обучение

Проблема: правки пользователя не закрепляются — те же ошибки повторяются.

1. **Захват правок как памяти фидбэка.** Когда пользователь поправляет агента
   («не так, надо вот так» / правит таблицу / отклоняет результат) — записать
   feedback-факт, привязанный к интенту/триггеру (через `orchestrator_memory` +
   `MemoryFact`), и учитывать его в будущем роутинге/построении спеки.
2. **Промоушн исправленных успешных ходов в рецепты.** Усилить `recipes.py`: ход,
   доведённый до «правильно» после правок, становится draft→active рецептом с
   корректным триггером; повтор похожего запроса воспроизводится без ошибки.
3. **Связать `orchestrator_memory` успех-skill с роутингом** (приоритет проверенных
   путей).
4. Метрика: частота повторения той же ошибки (должна падать), рост активаций
   рецептов, доля ходов без правок.

Файлы: `app/ai/orchestrator_memory.py`, `app/ai/recipes.py`, `app/ai/skill_evolver.py`,
`app/api/memory.py` (pin/feedback).

## Порядок и проверка
Ф1 (прямой удар по «делает не то») → Ф2 (причина на входе) → Ф3 (чтобы не
повторялось). После каждой фазы — пересборка прод-стека (`infra/`, prod-оверлей),
живой прогон реальных сценариев с контролем вывода на Рабочий стол, unit-тесты,
сравнение метрики до/после. Коммит по фазам в ветке `feat/reranker-llm-projects-objects`
(или новой `feat/agent-smartening`).
